# 内参文档排版质检指南

> 实战积累：当拷问框架应用于正式内参（内部参考）类文档的排版与格式审查时，需要关注的领域特有规范。

## 适用场景

- 报送领导/决策层的内部参考文章
- 政策研究报告、行业分析专报
- 需要正式排版的学术或调研报告

## 一、标题规范

内参标题需要**冲击力**——让读者（领导/决策者）看一眼就感知到"这事重要，值得细看"。

| 维度 | 好标题 | 差标题 |
|------|--------|--------|
| **句式** | 主谓宾结构 + 结论前置 | 偏正短语、学术定语过长 |
| **示例** | "95家SPV·千亿兜底——三层架构如何让房企资金'合法抽逃'" | "防范化解房市风险：房企系统性抽逃项目资金致基层政府兜底——机制溯源与制度解法" |
| **原则** | 一句话亮出核心矛盾和冲击力数字 | 面面俱到、像论文标题 |

## 二、头部摘要区

内参文档开头应有**内容提要**，格式如下：

```
> **内容提要**：
> 一句话点明问题（本文以……为实证基础，揭示……的核心困境）
>
> **核心发现**：① … ② … ③ …
> **政策建议**：……（三个方向+可行性分级）
> **关键词**：……（5-7个核心概念，用·分隔）
```

注意事项：
- 仅保留"内容提要"和"编制日期"两行，不写"课题来源""实证样本""法理依据"等学术框架
- 编制日期后附带一句话说明本报告的实证基础和数据来源

## 三、禁止使用的格式

### ❌ 不要用代码块（` ``` ` 或 `<pre>`）

所有架构图、流程图、数据结构：
- 用 **HTML架构图**（huashu-design 或 architecture-diagram skill 制作）
- 截图后作为图片插入文档
- 图片下方加图注："图1：……"
- 代码块在正式文档中显得粗糙且不可编辑

### ❌ 不要用 blockquote（`>` 竖线）引用法条

法律条文引用应使用正式格式：
- 方案A：缩进段落（两侧缩进+灰色文字）
- 方案B：直接引用，书名号明确出处
- 不要用 `>` 竖线，那在正式排版中属于"邮件引用"风格

### ❌ 不要用 callout/说明框标注"关键区分"类内容

如果需要在正文中插入区分性说明（如"资金归集 vs 资金抽逃"）：
- 揉入正文作为独立段落，用"值得指出的是……"自然过渡
- 不要在正文中单独标注"关键区分""案例说明"等标题
- 区分标题在论文/内参中是出戏的——把它变成正文的自然陈述

### ❌ 不要用"详情见独立文档"类指引

正式文档中不应出现"详见XX文档"的指引——要么把内容搬进文档，要么用附录或标注引用。

### ❌ 不要用 emoji 标记作为结构化元素

🟡、🔴 等 emoji 在正式文档中降低严肃感。可以用文字标签替代：
- 好：**立即可行** / **中期可行** / **长期目标**
- 差：🟡 可行性评估

### ❌ 减少加粗和强调（**text**）

正式内参中，加粗应极为克制——仅用于核心结论（一句话）。不要给每个术语、每条建议加粗。

## 四、排版检查清单

| 检查项 | 标准 |
|--------|------|
| 标题 | 一句话亮结论+数字冲击，无学术定语 |
| 摘要 | 内容提要+核心发现+政策建议+关键词 |
| 编制信息 | 仅日期+一句话数据来源 |
| 代码块 | 全部替换为架构图截图 |
| blockquote引用 | 改为缩进段落或直接引用 |
| callout/说明框 | 融入正文 |
| "详见"链接 | 改为直接引用或附录 |
| emoji标记 | 替换为文字标签 |
| 加粗（**） | 极度克制 |
| 图表编号 | 图1、图2连续编号+图注 |
| 可行性评估 | 子标题+文字清单，非blockquote |
| 免责声明 | 一句话精炼 |

## 五、实操记忆

用户 BINBIN 的内参排版偏好总结：
1. "让领导看一眼就觉得这事重要"——标题必须有冲击力
2. "尽可能精炼"——头部信息越少越好
3. "论文里哪能加个说明框"——分类标签融入正文
4. "正式文章不能用代码块"——HTML截图替代
5. "正式排版不这么用"——blockquote/callout/output 格式都是非正式的
