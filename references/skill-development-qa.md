# 方法论/技能方案质检指南

> 用拷问框架质检"方法论设计方案"的专用分支。
> 沉淀自 2026-05-27 思维工具链方案 v1→v2 迭代实战。

## 适用场景

当需要对一个**方法论/工具链/技能体系的设计方案**做质量检验时使用。

区别于内容/文章质检——这里拷问的对象不是孤立的文章，而是一套"可以被反复调用的思维工具"的设计。

## 核心差异点

| 维度 | 内容质检 | 方法论设计质检 |
|:----|:--------|:--------------|
| 对象 | 单篇内容 | 技能/Skill/工具链 |
| 检验重点 | 逻辑漏洞、论据可靠性 | 边界清晰度、技能间协作、用户认知负载 |
| 最致命的失败 | 论点被推翻 | 用户记不住、不愿用 |
| 质检标准 | 论点自洽 | 一句话能说清定位 |

## 实战校验清单（2026-05-27 验证）

### 必须回答的 6 个问题

1. **边界清晰吗？** —— 和其他技能/方法论的区分是什么？用户会困惑"我该用A还是B"吗？
2. **认知负载可控吗？** —— 总技能数是否超过5个？用户需要记多少东西才能用？
3. **有触发条件吗？** —— 用户说什么话时会自动唤起这个技能？触发词是否具体（不是"帮我分析"这种泛词）？
4. **协作链完整吗？** —— 上游产出能在下游直接消费吗？还是需要用户手动翻译？
5. **是独立存在的吗？** —— 删掉这个技能，其他技能还能正常运转吗？如果不能，是不是应该合并？
6. **命名有记忆点吗？** —— 用户提到它时，能用两个字的名字指代吗？

### 实战发现的典型问题

| 问题 | 发现方式 | 修复 |
|:----|:--------|:----|
| 钩玄vs撮要边界重叠 | 问题1 | 砍撮要，合并入钩玄作为会议分支 |
| 5+1个技能认知负载过高 | 问题2 | 压缩到4核心 |
| 纲目vs执要串行逻辑不成立 | 问题1+4 | 改为二选一分支 |
| 无触发词、无入口规则 | 问题3 | 每个skill增加触发词列表 |

## 输出格式偏好

用户偏好结构化诊断：表格+三色标记（🔴🟡🟢），一个问题一段。
不要冗长解释，直接结论+行动指引。

### 报告结构（2026-05-27 实战确认）

整份质检报告按 **上→下** 分层：

```
┌─ 建议与意见汇总（最上方）───┐
│ 🔴 必须修（P0，2-4条）      │
│ 🟡 建议优化（P1，2-4条）    │
│ 🟢 已经做对（确认项）       │
└────────────────────────────┘
┌─ 详细质检内容（下方）───────┐
│ 每个测试对象独立章节        │
│ 八维度逐项分析             │
│ 综合评级汇总表             │
└────────────────────────────┘
```

- 用户先看"建议与意见"决定是否行动
- 需要深入确认时，在下方"详细质检"中找对应条目
- 避免把行动建议和详细分析混在一起
